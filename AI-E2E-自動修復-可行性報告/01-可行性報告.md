# AI 開發後自動 E2E 驗證與自動修復可行性報告

## 1. 結論

結論是可行，但不能只靠 agent 自己打開頁面、看起來像成功就算成功。

穩定做法必須拆成三層：

1. 驗證層：Playwright 負責真實 pass/fail
2. 修復層：Qwen Code CLI 或 OpenCode CLI 負責讀證據後修補
3. 編排層：Shell 或 CI 負責控制輪次、保存工件、決定停止

## 2. 什麼情況適合導入

適合以下專案：

- 已有基本 E2E 測試，至少能跑 smoke
- AI CLI 可以在目前環境非互動執行
- 可以限制 AI 只修改白名單目錄
- 可以接受失敗時留下一批報告與日誌供後續分析

不適合以下專案：

- 根本沒有可重複執行的測試
- 每次啟動都高度依賴人工登入或人工驗證碼
- 目前連啟動方式都不穩定
- 無法接受 AI 自動修改程式碼

## 3. 建議架構

```text
AI 完成開發
  -> 啟動 app
  -> 跑 lint / typecheck / unit
  -> 跑 smoke E2E
  -> 跑 critical path E2E
  -> 若失敗，保存 logs / trace / screenshot / report
  -> 呼叫 AI CLI 讀證據並做最小修補
  -> 只重跑必要檢查
  -> 成功則結束
  -> 超過上限則輸出 triage report
```

## 4. 主要收益

- 把「修復前分析」標準化，不再完全靠人工看 log
- 把「修完有沒有真的好」交給測試框架，不靠主觀判斷
- 能在 CI 或本地重複運作，而不是一次性操作
- 適合把多數低風險、局部性的回歸問題自動處理掉

## 5. 主要風險

### 5.1 無限修補

典型症狀：

- 每輪都改一點，但沒有接近根因
- 一直在測試檔與 UI 細節間來回修改

控制方式：

- 設最大輪數
- 記錄每輪分類與根因
- 同類錯誤連續重複時直接停止

### 5.2 錯把環境問題當程式缺陷

典型症狀：

- 服務沒起來、port 衝突、env 漏掉
- AI 卻去改頁面邏輯或測試斷言

控制方式：

- 先做健康檢查
- Prompt 強制先分類
- 環境問題直接停止，不進入亂修

### 5.3 為了通過而弱化測試

典型症狀：

- 刪 assertion
- 改成過度寬鬆的等待條件
- 跳過關鍵測試

控制方式：

- Prompt 明確禁止弱化關鍵斷言
- 限制只做最小可辯護修補
- 要求輸出「為什麼這個修補是最小變更」

## 6. 最務實的導入順序

1. 先在本地跑通快速守門加 smoke 自修
2. 再補 critical path
3. 最後才接到 CI

先求可控，再求全自動。這樣比一開始就全量自動化穩得多。
