# AI 開發後自動 E2E 驗證與自動修復可行性報告

## 1. 報告目的

本報告評估以下目標是否可行，並提供可落地方案：

- AI agent 完成開發後，自動啟動應用
- 自動開網頁執行 end-to-end 測試
- 若測試失敗，agent 自動分析並修復
- 修復後自動重跑測試
- 在人不介入的情況下，重複執行直到成功或達到停止條件

結論：可行，但前提是要把「驗證」、「修復」、「編排控制」拆成三層，而不是只靠 agent 自己打開瀏覽器點幾下。

## 2. 結論摘要

### 2.1 可行性結論

此方案具備高可行性，適合以下情境：

- AI 已能穩定改 code、執行 shell、讀測試結果
- 專案已有或可以補上 Playwright E2E 測試
- 可以接受 agent 在受控範圍內自動修改程式碼
- 可以定義明確的停止條件與可修改檔案白名單

### 2.2 不建議的做法

不建議把流程設計成：

- agent 開網頁
- 自己判斷畫面看起來像成功
- 自己宣告任務完成

這種方式容易出現：

- 誤判成功
- 跳過邊界情境
- 無法穩定重現
- 出錯時沒有足夠工件讓 agent 分析

### 2.3 建議架構

建議採用三層架構：

1. 驗證層：`Playwright`
2. 修復層：`Qwen Code CLI` 或 `OpenCode CLI`
3. 編排層：`bash script`、`Makefile` 或 `CI workflow`

核心責任如下：

- Playwright：負責唯一的 pass/fail 判定
- AI CLI：負責讀取失敗資訊、修改 code、重新執行命令
- 外層 orchestrator：負責限制最大重試次數、保留 artifacts、決定何時停止

## 3. 問題定義

你的真實需求不是「讓 agent 幫我測一下」，而是：

- 建立一條無人值守的驗證與修復流水線
- 讓 AI 自己處理大部分失敗
- 最終只在真正卡住時才把結果丟給人看

這個問題本質上屬於「autonomous repair loop」，不是單純的 browser automation。

## 4. 方案選項

## 4.1 方案 A：Agent 直接開網頁手動試跑

### 描述

讓 agent 透過 browser automation 技能或內建瀏覽器能力，直接打開站點、點按鈕、檢查是否出現某些元素。

### 優點

- 上手快
- 不需要先寫完整測試框架
- 適合做一次性驗證或探索性檢查

### 缺點

- 驗證結果不夠穩定
- 難以做回歸測試
- 缺乏標準化報告與 artifacts
- 不利於 CI 導入
- agent 容易誤判

### 結論

只能當輔助，不適合當主方案。

## 4.2 方案 B：Playwright 驗證，Agent 修復

### 描述

用 Playwright 執行標準化 E2E，AI CLI 只讀取失敗結果並修復。

### 優點

- 判定標準明確
- 可回放
- 可重現
- 易於 CI 化
- 失敗定位相對穩定

### 缺點

- 需要先建立 smoke 與 critical path 測試
- 專案前期要投入一點測試基礎建設

### 結論

這是最平衡、最建議的方案。

## 4.3 方案 C：純 CI 自動跑，不做 AI 自動修復

### 描述

只在 CI 內跑 E2E，失敗後通知人處理。

### 優點

- 實作簡單
- 風險低

### 缺點

- 沒有達到無人介入目標
- 修復成本仍在人身上

### 結論

可以當第一階段，但不是你的最終目標。

## 5. 建議落地架構

## 5.1 推薦架構圖

```text
AI 開發完成
  -> 啟動應用與依賴服務
  -> 跑 lint / typecheck / unit
  -> 跑 Playwright smoke
  -> 跑 Playwright critical path
  -> 若失敗，保存 report / trace / screenshot / logs
  -> 呼叫 Qwen Code CLI 或 OpenCode CLI 分析並修復
  -> 只重跑失敗測試或重跑 smoke
  -> 成功則結束
  -> 超過重試上限則輸出 triage report 並停止
```

## 5.2 各層責任

### 驗證層

建議使用 Playwright，因為它適合：

- 瀏覽器操作
- trace、screenshot、video 蒐證
- HTML/JSON 報告
- smoke 與 critical path 分層
- CI 執行

### 修復層

可選：

- `Qwen Code CLI`
- `OpenCode CLI`

角色定位：

- 讀取測試失敗訊息
- 讀取 artifacts 摘要
- 判斷錯誤類型
- 修改最小範圍的 code
- 重跑命令

### 編排層

建議用 shell script 或 CI workflow 明確控制：

- 最大重試次數
- 可修改檔案白名單
- 哪些 command 可以執行
- 哪些情況必須立刻停止

## 6. 成功關鍵

## 6.1 先有最小可用測試集

至少要先有：

- smoke tests
- critical path tests

沒有測試基準，agent 再強也無法穩定驗證「真的修好」。

## 6.2 artifacts 必須完整

每次失敗至少保留：

- Playwright HTML report
- trace
- screenshot
- video
- app logs
- test stdout/stderr

否則 agent 會依靠猜測修 code，品質會很差。

## 6.3 停止條件必須先定義

至少定義以下條件：

- 最多修復輪數，例如 3 到 5 輪
- 同一錯誤不得無限重試
- 若出現 migration、schema、infra 類變更則停止
- 若需要新增 secrets 或修改 production config 則停止

## 6.4 可修改範圍必須白名單化

例如：

- 可改：`src/`、`app/`、`tests/e2e/`、`playwright.config.*`
- 不可改：`infra/`、`deploy/`、`.github/secrets`、production env

## 7. 風險評估

## 7.1 主要風險

### 風險 1：無限修補

症狀：

- agent 不斷修改測試或 UI 細節
- 每輪都換一種寫法但沒有真正解決問題

對策：

- 限制最大輪數
- 若同一類錯誤重複出現兩次以上則停止

### 風險 2：把測試修到「會過」但功能沒修好

症狀：

- agent 直接放寬 assertions
- 跳過重要驗證

對策：

- 禁止 agent 任意刪減 critical assertions
- 對測試檔案修改設額外規則

### 風險 3：環境問題被誤判成程式碼問題

症狀：

- port 沒起來
- 測試資料不存在
- 第三方服務 timeout

對策：

- 先做 health check
- 明確區分 app bug、test data issue、env issue、flaky

### 風險 4：修復範圍過大

症狀：

- agent 同時修改多個不相關模組
- 修好 E2E 但造成其他回歸

對策：

- 限制單輪 diff 大小
- 每輪先跑快速守門再跑 E2E

## 8. 導入建議

## 8.1 第一階段

- 建 smoke tests
- 建本地自動修復腳本
- 先在個人開發環境運行

## 8.2 第二階段

- 加入 critical path E2E
- 導入 artifacts 保存
- 引入 triage report

## 8.3 第三階段

- 移到 CI
- 只允許在 staging 或 ephemeral env 自動修復
- 對主分支採較保守策略

## 9. 最終建議

若你的目標是「AI 開發完後，全自動驗證、失敗自修、重跑直到成功」，最佳落地方式是：

- 用 Playwright 當唯一驗證標準
- 用 Qwen Code CLI 或 OpenCode CLI 當修復代理
- 用 shell script 或 CI workflow 控制重試、蒐證與退出條件

這條路線具備高可行性，且能在不需要人持續盯著的前提下，把大多數回歸錯誤攔下並自動處理。

不過它不是「agent 自己會測」這麼簡單，而是要建立一條受控的工程化閉環。
