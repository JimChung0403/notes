AI 開發工作流 Playbook

以 OpenSpec 為規格主源、 Superpowers 為執行紀律的實務操作指南。 涵蓋八種 Brownfield 開發情境，每種情境對應具體指令序列。

適用：既有專案導入 AI 輔助 涵蓋情境：8 種 七階段：讀懂 / 分析 / 設計 / 實作 / 測試 / 審查 / 回寫

00前提設定

0.1 工具安裝

# 初始化 OpenSpec（在你的 repo 根目錄執行）
npx @fission-ai/openspec init

# 安裝 Superpowers（在 Claude Code 裡輸入）
/plugin install superpowers@claude-plugins-official

0.2 CLAUDE.md 必備三條規則

OpenSpec 跟 Superpowers 不會自動串接 — 必須在 CLAUDE.md 明寫下面三條，否則 TDD 不會被觸發。

## AI Workflow Rules

# Rule 1: OpenSpec 是唯一規格源
When /opsx:apply runs, MANDATORILY use these Superpowers skills:
  - test-driven-development (RED-GREEN-REFACTOR)
  - verification-before-completion
  - subagent-driven-development for code review

# Rule 2: 禁用 Superpowers 的 writing-plans
DISABLE Superpowers writing-plans skill.
OpenSpec's tasks.md is the single source of truth.

# Rule 3: Archive 是硬閘門
Every feature/fix must end with /opsx:archive before PR merge.
No exceptions. Enforce via pre-merge Git hook.

0.4 Brownfield 首次導入：Codebase 盤點

既有專案第一次啟用 AI workflow 時，跑兩個一次性的盤點 command，產生長駐文件供後續每個 feature 參考。這是 Brownfield 專案不可跳過的前置步驟。

指令 A：/gsd:map-codebase（強烈建議）

GSD 的 codebase-mapper 可當獨立 skill 使用（不必裝完整 GSD）。它 spawn 4 個 parallel subagents，各自聚焦產出以下文件：

Focus產出文件內容
techSTACK.md
INTEGRATIONS.md		技術棧 + 外部整合清單
archARCHITECTURE.md
STRUCTURE.md		架構圖 + 檔案結構 + 模組關聯
qualityCONVENTIONS.md
TESTING.md		命名慣例 + 測試 pattern（附實際程式碼範例）
concernsCONCERNS.md技術債 + 已知問題 + 風險點

產出位置：.gsd/ 目錄。每份文件會包含實際檔案路徑（backtick 格式），方便 Claude 直接跳到 code。

# 在 Claude Code 裡執行
/gsd:map-codebase

# 或指定 focus（單獨 refresh 某一塊）
/gsd:map-codebase --focus arch

指令 B：/opsx:onboard（OpenSpec 內建，選配）

為既有 code 反推出初始規格，產在 openspec/specs/。補足 GSD 沒有的「規格層」視角。

/opsx:onboard
建議做法兩個都跑。/gsd:map-codebase 給你「code 長怎樣」的全景；/opsx:onboard 給你「既有規格從哪繼承」的起點。兩者互補。

更新頻率

  • 重大架構變更時（新增 / 刪除服務、框架升級、DB schema 大改）
  • 每 3 個月 refresh 一次（設日曆提醒）
  • 新人 onboarding 前確認是最新版本
  • Sprint retrospective 時列為檢查項
為什麼不可省每個 feature 的 Step 1（讀懂）會引用這 7+ 份檔案，讓 OpenSpec 的 /opsx:propose 不會憑空設計跟既有架構打架的 schema / API。跳過這步 → AI 產出的 migration 很可能撞既有 table。

0.5 設定 openspec/config.yaml

OpenSpec 初始化（npx @fission-ai/openspec init）會產一份 openspec/config.yaml。這是 OpenSpec 專屬的控制點，定義「每次跑 /opsx:* 自動注入什麼 context 跟套什麼規則」。把 0.4 產出的 .gsd/ maps 連結進來，讓 OpenSpec 指令永遠看到它們。

三個欄位

欄位作用上限
schema預設 schema（省掉每次 --schema flag）—
context全域 context，注入每個 /opsx:* 指令50KB
rulesper-artifact 規則（proposal / design / specs / tasks）—

完整樣板（可直接抄用）

schema: spec-driven

context: |
  # Project Overview
  Tech stack: TypeScript, React, Node.js, PostgreSQL
  API style: RESTful, documented in docs/api.md
  Testing: Jest + React Testing Library

  # Codebase Maps (MUST consult before any design decision)
  - .gsd/ARCHITECTURE.md  (API & data flow style)
  - .gsd/STRUCTURE.md     (module layout)
  - .gsd/CONVENTIONS.md   (naming, linting)
  - .gsd/STACK.md         (approved libraries)
  - .gsd/TESTING.md       (test patterns)
  - .gsd/INTEGRATIONS.md  (external services)
  - .gsd/CONCERNS.md      (known pitfalls — avoid)

  # Incoming Specs
  PM specs arrive at: docs/incoming-specs/{feature}.md
  Always reference the PM spec when generating proposal.md.

  # Governance
  Respect CONSTITUTION.md at repo root (if exists).

rules:
  proposal:
    - Include rollback plan for any DB or API change
    - Identify affected teams / services
    - Reference .gsd/CONCERNS.md for known pitfalls

  design:
    - API endpoints MUST follow .gsd/ARCHITECTURE.md patterns
    - Deviations MUST be labeled "【AI 建議】" with rationale
    - Schema changes MUST align with .gsd/STRUCTURE.md
    - Library choices MUST come from .gsd/STACK.md (or propose addition)

  specs:
    - Use Given/When/Then format
    - Reference existing patterns before inventing new
    - Acceptance criteria must map to tasks.md checkboxes

  tasks:
    - Each task estimated 15–60 minutes
    - Each task references PM spec section OR .gsd/ file
    - Test-writing tasks follow .gsd/TESTING.md patterns

跟 CLAUDE.md 的分工

設定檔作用範圍負責放什麼
CLAUDE.mdClaude Code 全域 sessionTDD / archive 硬規則、@-references
openspec/config.yaml/opsx:* 指令執行時artifact 規則、OpenSpec 專屬 context
分工原則通用紀律放 CLAUDE.md；規格層規則放 config.yaml。例如「必跑 TDD」放 CLAUDE.md；「design.md 內 API 必須對齊 .gsd/ARCHITECTURE.md」放 config.yaml。

更新時機

  • 技術棧改變（新框架、換 DB、新 library）
  • .gsd/ maps refresh 後同步更新 context 裡的引用
  • 新增某類 artifact 的規則時
  • Sprint retrospective 發現某條規則無效或缺規則時
注意context 欄位硬上限 50KB。不要把 .gsd/ 檔案整個貼進來 —— 只放「該讀哪個檔」的指引，AI 會用 @-reference 實際去讀。貼整份檔 = 浪費 config 空間 + 每次變更要同步更新兩處。
三層護欄總覽
  1. CLAUDE.md @-references（Layer 1）— Claude Code 全域自動載入 .gsd/ 檔
  2. CLAUDE.md mandatory rules（Layer 2）— 強制 TDD / review / archive
  3. openspec/config.yaml（Layer 3）— OpenSpec 指令層 artifact 規則
三層疊加，彼此不知道對方存在，但都會被讀。

03分析階段：OpenSpec + Brainstorming 工具交接

OpenSpec 跟 Superpowers 的 brainstorming skill 不會自動串接 — 你得決定誰先誰後、誰質疑誰。以下三種模式按 PM spec 的清晰度選擇。

APropose 先，Brainstorming 後（質疑用）
適用PM spec 清楚 頻率90% 情境
用途Sanity check — 進實作前挑 OpenSpec 草稿的漏洞
brainstorming 角色Red team（挑毛病）

Step 1：OpenSpec 先產草稿

User: /opsx:propose "通知系統：支援 email + push，使用者可自訂偏好"

OpenSpec 產出:
  changes/notification-system/
    ├── proposal.md   ← 需求與 scope
    ├── design.md     ← 技術方案草稿
    ├── specs/        ← requirements
    └── tasks.md      ← 實作清單

Step 2：啟動 brainstorming 質疑草稿

User: 請啟動 Superpowers brainstorming skill，
      閱讀 changes/notification-system/design.md，
      用 Socratic method 挑戰裡面的假設。

      - 補上 retry 策略（exponential backoff, max 3 次）

  - 產 exploration notes

User: 啟動 brainstorming skill 一起探討

User: /opsx:propose "..." (基於 exploration 的結論)
重點對應 情境 7（Spike）。/opsx:explore + brainstorming 是雙人偵查隊。

三秒判斷法

PM spec 看完第一眼的感覺—— ├─ 「可以直接做」(清楚) → 模式 A：Propose 先，Brainstorm 質疑 ├─ 「還缺很多細節」(模糊) → 模式 B：Brainstorm 先，Propose 收尾 └─ 「不確定該不該做」(探索) → 模式 C：Explore + Brainstorm 交替

最常踩的坑

錯誤做法為什麼不好
兩個同時開Socratic 提問會跟 OpenSpec 產 artifact 的步驟混亂
brainstorming 跑完沒寫回 design.md對話結束後 artifact 沒更新 → design.md 跟討論脫節
在 /opsx:apply 之後才 brainstorming太晚了，已經在寫 code。質疑要在 design 階段
brainstorming 問題超過 10 個變成審問 PM。目的是解鎖共識，不是盤查
實務建議90% 情境用模式 A。直接 /opsx:propose 先產草稿，再啟 brainstorming 質疑 5–10 分鐘就好。只有 PM 明確說「我也不知道細節」才切模式 B。模式 C 通常只在 sprint planning 用到（對應情境 7 Spike）。

04Brownfield 開發情境

八種常見情境。每個情境會告訴你：何時適用、七階段怎麼對應指令、以及容易踩的坑。

01新 Sprint 起跑
耗時1–3h / sprint 頻率每 sprint 一次
何時適用每個 sprint 開始的規劃階段
關鍵風險Backlog 沒分類 → 小 bug 被當大 feature 做

這不是一個 feature 的 7 階段，而是把 backlog 分類、指派到下面的情境 2–8。

#步驟工具 / 指令產出
1盤點已 archived 的規格閱讀 openspec/specs/現狀認知
2PM 列出 sprint backlog— (人類作業)backlog.md
3逐項判斷情境類別參考下方決策樹每 item 貼標籤
4大 feature 預先 propose/opsx:propose "feature X"changes/{feature-x}/
5小 fix 先不 propose— (動工當下才跑)—
注意不要對所有 backlog item 都跑 /opsx:propose — 會產生一堆用不到的 change dir。只對 4 小時以上的 feature 預先 propose。
首次導入若是第一次在這個 repo 裝 OpenSpec，這時候跑 /opsx:onboard 為既有 codebase 反推出一份初始規格。只做一次。

情境判斷決策樹

Backlog item 是什麼？ ├─ Bug │ ├─ Production 宕機 / 資安事件 → 情境 5（Hotfix） │ └─ 一般 bug → 情境 4（Bug fix） ├─ 新功能 │ ├─ < 2 小時 → 情境 3（臨時小改動） │ └─ 4–16 小時 → 情境 2（Sprint 中新功能） ├─ 改進既有 code │ └─ 不改行為、改結構 → 情境 6（重構） ├─ 升級 │ └─ 依賴 / CVE / 安全 → 情境 8（依賴升級） └─ 可行性不確定 → 情境 7（Spike）
02Sprint 中新功能（4–16 小時）
耗時4–16h 頻率主流情境 風險跳階段
何時適用PM 寫好 spec 的正規 feature，會改 3+ 檔案
關鍵風險最常見 — 跳過某個階段，尤其是回寫
前置檢查開工前先確認 .gsd/ 目錄的 codebase maps 是最新版本（產出自 Section 0.4）。若 > 3 個月沒跑，先執行 /gsd:map-codebase refresh，再進 Step 1。

Step 1 起手式（PM spec 詳細度依賴流程）

PM 交付的 spec 通常有三種詳細度，對應不同起手法：

PM spec 詳細度Step 1 做法
完整（含 API + 完整 schema） 直接 /opsx:propose 引用 PM spec，規則寫「保留不變」
中等（鎖 scope + 提到部分資料表與欄位值）
← 最常見		 PM spec → docs/incoming-specs/
/opsx:propose 同時引用 PM spec 與 .gsd/ maps（見下方範例）
模糊（只有需求描述） 改用 模式 B：先 brainstorming 精煉，再 propose

中等詳細 spec 的 /opsx:propose 樣板（最常用）

/opsx:propose

Source material:
- PM spec: @docs/incoming-specs/{feature}.md
- Codebase context: @.gsd/ARCHITECTURE.md @.gsd/CONVENTIONS.md @.gsd/STACK.md @.gsd/STRUCTURE.md

規則：

【鎖定 - 不得改動】
1. scope、驗收條件 → 照 PM spec
2. PM 指定的既有資料表 / 欄位 / 值 → 保留
3. 遵守 .gsd/CONVENTIONS.md 的命名風格與 linting 規則

【開放 - AI 設計，但需標註】
4. API endpoint → 沿用 .gsd/ARCHITECTURE.md 的 API 風格設計，標 "【AI 建議】"
5. 新欄位 / 新資料表 → 對齊 .gsd/STRUCTURE.md 的模組劃分
6. 測試 → 照 .gsd/TESTING.md 的既有 pattern

【硬規則】
7. 使用既有表時，欄位型別必須對齊 .gsd/ 相關文件
8. 若踩到 .gsd/CONCERNS.md 提到的已知問題 → 在 design.md 標註處理方式
9. 模糊或衝突處 → 列在 Open Questions，不自行假設

✅ 適合❌ 不適合

  • 快速迭代勝於文件完整性

七階段對應（OpenSpec 指令 + 人工紀律）

階段OpenSpec 指令需人工補的部分
讀懂/opsx:propose 或 /opsx:explore—
分析OpenSpec 自動產 proposal.md + design.md 草稿人工 brainstorming / review design（建議 15–30 分）
設計design.md + specs/ + tasks.md—
實作/opsx:apply（工具只追 tasks.md 狀態）人工寫 code（或搭 Cursor / Copilot）
人工強制 TDD（pre-commit hook 擋沒 test 的 commit）
測試—（OpenSpec 不覆蓋）人工寫 test 全包：unit / integration / E2E
審查—（OpenSpec 不覆蓋）人工 GitHub PR review
可選：用其他 AI tool 做 code review（Cursor AI / Codex）
回寫/opsx:verify → /opsx:archive—
關鍵補洞沒有 Superpowers → 測試 + 審查變成團隊紀律問題，工具管不到。三個補救措施必做：
  1. Pre-commit hook 擋無 test 的 commit
  2. CI 要求 test coverage ≥ 某閾值
  3. GitHub branch protection 強制 PR review

08常見困惑

Q1. 為什麼不能跳過 /opsx:archive？
沒 archive = changes/{name}/ 這個目錄永遠存在，規格永遠沒合進永久 specs/。下次 PM 來看 spec 會看到過時版本，下次 AI 讀 specs 做決策會基於舊規格。這會放大成「為什麼 AI 寫出來的 code 跟現況對不上」。
Q2. TDD 真的每次都要寫 test 嗎？連 log 都要？
是。加 log 的 test 就是「呼叫後確認 log 有某字串」，寫起來 5 秒。一旦開始豁免，團隊會越豁越多，回頭發現 70% 功能沒測試。
Q3. Hotfix 階段二沒補怎麼辦？
這是最常見的長期風險。建議：
  • 設 CI 檢查 hotfix/* branch 24 小時內必須有配對的 change/ 目錄
  • Sprint retrospective 必列「hotfix 補寫狀態」
Q4. OpenSpec 的 changes/ 目錄跟 Git branch 的關係？
每個 changes/{name}/ 對應一個 feature branch：
  • Branch name: feat/{change-name} / fix/{change-name}
  • PR 合併後 → /opsx:archive → changes/archive/{name}/
  • 目錄名稱與 branch 名稱一致，方便追溯
Q5. 可以跳過 subagent review 只做人工 review 嗎？
可以，但不建議。Superpowers 的 subagent-driven-development 是用獨立 subagent 讀 diff 找問題 — 它跟主 session 不共享 context，等於第二雙眼睛。人類 reviewer 容易 tunnel vision，subagent review 是便宜的保險。
Q6. 團隊有人不想用，可以局部採用嗎？
可以，但要畫清界線：
  • 必做：使用 AI 開發的檔案必須跑完 /opsx:archive + TDD
  • 可選：純人工寫的檔案可以不進 OpenSpec flow
  • 禁止：同一 PR 裡一部分走 OpenSpec 一部分不走 — 會造成規格斷層
Q7. 情境 2 走完要多久？值得嗎？
熟練後一個 8 小時的 feature 從 /opsx:propose 到 /opsx:archive 全程約 8–12 小時（多出的 50% 是 ceremony 成本）。ROI 體現在：
  • 規格永久可查 → 半年後新人能看懂
  • TDD 減少回歸 bug → 減少情境 4 的總量
  • Archive 紀錄決策 → 減少「為什麼當初這樣設計」的考古成本

# AI-Assisted Development Rules

## Tool Stack
- OpenSpec: Source of truth for specifications (propose/apply/archive)
- Superpowers: Execution discipline (TDD/review/verification)

## Mandatory Chain Rules

### Rule 1: OpenSpec Drives Specs
- OpenSpec's `tasks.md` is the single source of truth for implementation plans.
- DISABLE Superpowers `writing-plans` skill. Do not create parallel plans.

### Rule 2: /opsx:apply Triggers Superpowers Discipline
When /opsx:apply runs, mandatorily activate:
- test-driven-development (RED → GREEN → REFACTOR)
- verification-before-completion (before marking task done)
- subagent-driven-development (for code review)

### Rule 3: TDD Semantics by Scenario
- Feature / Bug fix: Write failing test FIRST, then code.
- Refactor: Do NOT add or modify tests. Only change code while keeping tests green.
- Spike: TDD exempt. Code is disposable.

### Rule 4: Archive Is a Hard Gate
- No PR may merge to `main` with a non-archived `changes/` directory.
- Enforce via pre-merge Git hook: `./scripts/check-no-pending-changes.sh`.

### Rule 5: Hotfix Follow-up
- `hotfix/*` branches must have a matching archived change within 24 hours.
- Enforced by CI scheduled check.
